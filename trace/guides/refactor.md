# GUIDE: Refactor

## Цель
Улучшить структуру кода без изменения наблюдаемого внешнего поведения.

## Обязательный порядок действий
1. Проверить, что задача действительно является refactor, а не feature или behavior change.
2. Прочитать purpose, architecture map и relevant contracts.
3. Явно перечислить, что должно остаться неизменным.
4. Сделать маленькие и проверяемые шаги.
5. Сохранить совместимость контрактов.
6. Прогнать тесты.
7. Обновить документацию при необходимости.

## Нельзя
- Скрытно менять бизнес-логику под видом рефакторинга.
- Менять API без подтверждения.
- Переносить ответственность между модулями без решения человека.

## Definition of Done
- внешнее поведение не изменилось;
- код стал проще, чище или устойчивее;
- тесты подтверждают отсутствие регрессии.
