# GUIDE: Bugfix

## Цель
Исправить причину дефекта минимально достаточным изменением, не создавая архитектурный дрейф и не нарушая существующие инварианты.

## Обязательный порядок действий
1. Понять симптом и ожидаемое корректное поведение.
2. Прочитать relevant purpose, contract, stack guardrails и repo map.
3. Определить истинную границу дефекта.
4. Проверить, затрагивает ли исправление API, БД, контракты, инварианты.
5. Если да — показать план и ждать подтверждения.
6. Если нет — внести минимально достаточный патч.
7. Добавить regression test.
8. Обновить контракт или docs, если изменена интерпретация поведения.
9. Вернуть краткий отчёт.

## Нельзя
- Чинить симптом вместо причины, если причина локализуема.
- Переписывать модуль целиком без необходимости.
- Менять публичное поведение без явного указания.
- Оставлять изменение без regression test, если дефект воспроизводим.

## Definition of Done
- дефект исправлен;
- есть regression test;
- не нарушены инварианты;
- изменения локальны и объяснимы;
- связанные артефакты синхронизированы.
