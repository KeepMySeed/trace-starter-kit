# GUIDE: Feature Delivery

## Цель
Добавить новую возможность в рамках существующего архитектурного замысла, сохранив целостность модулей, контрактов и инвариантов.

## Обязательный порядок действий
1. Прочитать purpose, architecture map, stack guardrails и relevant contracts.
2. Выделить целевой use case и границы изменения.
3. Определить, нужен ли новый контракт или расширение существующего.
4. Представить план, если затрагиваются жёлтые зоны.
5. Реализовать минимально достаточный набор изменений.
6. Добавить тесты на новый сценарий.
7. Обновить документацию и traceability.
8. Вернуть implementation report.

## Важно
- Новая функция должна быть связана с конкретным use case.
- Не допускается скрытое расширение ответственности модуля.
- Любой новый внешний контракт должен быть описан явно.

## Definition of Done
- use case реализован;
- тесты добавлены;
- контракт описан или обновлён;
- архитектурные границы соблюдены;
- изменения объяснимы и проверяемы.
