# TRACE Starter Kit

Готовый starter-kit для внедрения **TRACE (Traceable Retrieval-Anchored Context Engineering)** в репозиторий с AI-ассистированной разработкой.

Этот шаблон нужен для того, чтобы AI-агент работал не по хаотичному контексту, а по явным артефактам:

* замыслу системы,
* ограничениям по стеку,
* карте архитектуры,
* контрактам модулей,
* гайдам по типам задач,
* правилам автономии,
* трассируемости,
* верификации изменений.

# Быстрый старт

## 1\. Для каких проектов это подходит

TRACE особенно полезен, если:

* репозиторий уже больше 30–50 файлов;
* код регулярно меняют AI-агенты;
* в проекте есть сложная бизнес-логика;
* баги стоят дорого;
* важны предсказуемость, повторяемость и audit trail;
* несколько разработчиков и/или агентов работают над одним кодом.

Упрощённую версию TRACE можно использовать и в меньших проектах, но в маленьком прототипе не нужно насаждать всю методологию целиком.

## 2\. Что сделать сразу после копирования starter-kit

1. Заполнить `trace/intent/purpose.yaml`.
2. Заполнить `trace/intent/stack\_guardrails.yaml`.
3. Описать 5–10 главных модулей в `trace/architecture/architecture\_map.yaml`.
4. Создать контракты только для критичных модулей.
5. Прочитать и адаптировать `AGENT.md` под ваш проект.
6. Подключить генерацию `repo\_map.json`.
7. Использовать гайды из `/trace/guides` при постановке задач агенту.

# Базовый рабочий цикл

## Постановка задачи

Перед началом задачи агент должен получить:

* цель;
* ограничения;
* список запрещённых изменений;
* релевантные артефакты TRACE;
* ожидаемые проверки;
* уровень автономии.

Пример:

```text
Цель: исправить дублирование invoice events при повторной обработке webhook.

Прочитай:
- /trace/intent/purpose.yaml
- /trace/contracts/billing.payment.apply.yaml
- /trace/guides/bugfix.md
- релевантный фрагмент /trace/graph/repo\_map.json

Ограничения:
- не менять public API
- не менять схему БД
- не вводить новые зависимости

Ожидаемый результат:
- локальный патч
- тест на регрессию
- обновление контракта при необходимости
- краткий отчёт по шаблону implementation\_report.md
```

## Выполнение задачи агентом

Правильный порядок работы:

1. Понять цель и ограничения.
2. Прочитать релевантный контракт.
3. Собрать минимально достаточный контекст.
4. Показать план, если задача затрагивает жёлтую зону.
5. Внести минимально достаточное изменение.
6. Прогнать проверки.
7. Обновить контракт, документацию и трассируемость.
8. Вернуть отчёт.

# Уровни автономии

## Зелёная зона

Агент может делать сам:

* локальные исправления логики внутри контракта;
* добавление тестов;
* улучшение сообщений об ошибках;
* синхронизацию документации;
* рефакторинг без изменения внешнего поведения.

## Жёлтая зона

Нужно подтверждение человека:

* изменение public API;
* изменение схемы БД;
* изменение межмодульных контрактов;
* изменение формата внешних событий;
* новая зависимость;
* изменение логики, влияющей на инварианты.

## Красная зона

Нельзя без отдельного решения:

* изменение архитектурных границ модулей;
* ослабление требований безопасности;
* удаление трассируемости и наблюдаемости;
* изменение критичных бизнес-инвариантов.

# Семантические якоря в коде

TRACE не требует размечать весь код. Якоря нужны только там, где это даёт пользу:

* критичные модули;
* сложная бизнес-логика;
* зоны с частыми доработками;
* интеграционные boundary points.

Пример:

```ts
// <TRACE:MODULE id="billing.invoice" contract="CONTRACT.billing.invoice.create">
// <TRACE:FUNCTION id="billing.invoice.create" req="REQ-12,UC-4">
// <TRACE:BLOCK id="validate\_input" purpose="Validate invoice payload">
function createInvoice(input: CreateInvoiceInput): CreateInvoiceResult {
  // ...
}
// </TRACE:BLOCK>
// </TRACE:FUNCTION>
// </TRACE:MODULE>
```
